# C++ 必会知识：从对象模型出发

“必须会”的判断标准不是语法是否常见，而是：不会它是否容易写出未定义行为、资源泄漏、错误接口或无法维护的程序。下面各项应能解释原理，并能独立写出小例子。

## 1. 编译、链接与 ODR

C++ 源文件先各自编译为目标文件，再由链接器合并。一个 `.cpp` 连同它展开后的头文件称为一个翻译单元。

- 声明告诉编译器“名字及类型是什么”；定义提供实体。
- 普通函数通常在头文件声明、在 `.cpp` 定义。
- 模板定义必须在实例化位置可见，通常完整放在头文件。
- 头文件使用 include guard 或 `#pragma once`。
- ODR（单一定义规则）要求非 `inline` 实体在整个程序中只有一个定义。

这解释了为什么“语法检查通过”仍可能链接失败，也解释了 chapter-14 原先把模板成员放进 `.cpp` 后出现的未定义符号。

## 2. 类型、初始化与转换

对象是“一段存储 + 一个类型 + 一段生命周期”。类型决定有效操作和解释字节的方式。

- 优先使用花括号或明确的构造初始化，避免未初始化的内置类型。
- `auto` 保留值类别推导，但会丢掉顶层 `const` 和引用；需要时写 `const auto&`。
- 有符号与无符号整数混算可能产生意外的大正数。
- 使用 `static_cast` 表达可审查的显式转换；避免 C 风格强制转换。
- 浮点数通常不能用 `==` 判断计算结果是否“数学上相等”。

## 3. 生命周期、作用域与未定义行为

访问一个对象的前提是它的生命周期仍在持续。编译器假设程序没有未定义行为（UB），因此 UB 不能靠“这次运行正常”证明安全。

常见 UB：越界、解引用空指针或悬空指针、读取未初始化值、数据竞争、重复释放、除零，以及通过错误类型访问存储。

```cpp
const int* bad() {
    int local = 42;
    return &local; // 返回后 local 生命周期结束，指针悬空
}
```

引用不是拥有者；指针也不自动表达所有权。接口必须让生命周期关系足够清楚。

## 4. RAII 与所有权

RAII 的推导很直接：栈展开一定会销毁已经构造成功的局部对象，因此把资源释放放进析构函数，就能覆盖正常返回和异常路径。

- 值对象与容器优先。
- 独占堆对象使用 `std::unique_ptr`。
- 确实存在共享生命周期时才使用 `std::shared_ptr`。
- 非拥有观察使用引用、指针、`std::span`（C++20）或迭代器。
- 文件、锁、线程等资源也应由对象管理，不只内存如此。

示例：[ownership.cpp](../examples/core-language/ownership.cpp)。

## 5. 值语义、复制、移动与 Rule of Zero

值语义意味着对象的复制结果彼此独立，移动则把可转移资源交给新对象。一个类若只由正确实现值语义的成员组成，编译器生成的复制、移动和析构通常正是所需行为，这就是 Rule of Zero。

如果类直接拥有裸资源，就必须认真处理析构、复制构造、复制赋值、移动构造和移动赋值（Rule of Five），还要保证自赋值和异常安全。通常更好的修复是把裸资源换成 `std::vector`、`std::string` 或智能指针。

示例：[value_semantics.cpp](../examples/core-language/value_semantics.cpp) 与 [`matrix.h`](../notes/books/essential-cpp/chapter-04/matrix.h)。

## 6. `const`、引用与函数接口

先决定函数是否需要所有权，再决定参数形式：

- 小型可复制输入：按值传递，如 `int`。
- 只读且复制昂贵：`const T&`。
- 必须修改调用者对象：`T&`。
- 参数可缺省为空：`T*` 或合适的可空包装类型。
- 需要取得所有权：按值接收所有权类型，再移动进成员。

返回局部值是安全且高效的，编译器可执行返回值优化。不要为了“避免复制”返回局部对象的引用。

## 7. 类、不变量与多态

类的价值是维持不变量，而不只是把字段放在一起。构造完成后对象就应有效；每个公开操作都应保持不变量。

运行时多态需要基类引用或指针以及虚函数：

- 多态基类析构函数必须是 `virtual`。
- 派生类重写函数使用 `override`，让编译器检查签名。
- 避免对象切片：按值把派生对象复制到基类对象会丢失派生部分。
- 组合通常比继承耦合更低；继承应表达可替换的 “is-a” 关系。

## 8. 标准库：容器、迭代器与算法

应熟练使用 `vector`、`string`、`array`、`unordered_map`、`map`，理解它们的复杂度和迭代器失效规则。

算法把“遍历策略”与“元素操作”分开。优先用范围 `for` 和 `<algorithm>`，减少手写下标、边界和循环状态。

```cpp
std::sort(values.begin(), values.end());
const auto found = std::find(values.begin(), values.end(), target);
```

示例：[algorithms.cpp](../examples/core-language/algorithms.cpp)。

## 9. 模板与泛型

模板是在编译期根据类型生成代码。一个可用的泛型算法只依赖它真正需要的操作，而不依赖具体类型。

- 理解模板参数推导、实例化和错误发生的位置。
- 理解 `typename`、完美转发和类型特征的用途，不必滥用元编程。
- C++20 项目应使用 concept 表达约束；C++17 中至少通过命名和文档明确要求。
- 模板代码同样要遵守 ODR，定义通常位于头文件。

## 10. 错误处理与异常安全

错误处理首先区分：调用者可恢复的运行时错误、程序员违反前置条件，以及无法继续的系统故障。

- 可恢复失败可使用返回值、`std::optional`、错误码或异常。
- 构造函数无法返回错误码，异常适合报告构造失败。
- `assert` 用于检查程序员应保证的不变量，不用于处理用户输入。
- 异常按 `const&` 捕获，避免复制和切片。
- 修改多个状态时，先构造替代值，成功后再提交，以提供强异常保证。

示例：[error_handling.cpp](../examples/core-language/error_handling.cpp)。

## 11. 并发与内存模型

线程安全的第一原则是先减少共享可变状态。若两个线程无同步地访问同一对象，且至少一个访问是写操作，就发生数据竞争，程序具有未定义行为。

- 用 `std::mutex` 保护一组共同不变量，而不是零散字段。
- 用 `std::lock_guard` / `std::scoped_lock` 让解锁遵循 RAII。
- 条件变量等待必须带谓词，因为存在虚假唤醒。
- 原子操作只保证指定原子对象及所选内存序；它不是任意数据结构的自动线程安全开关。
- 线程对象析构前必须 `join` 或 `detach`；优先使用更高层任务抽象。

示例：[concurrency.cpp](../examples/core-language/concurrency.cpp)。

## 12. 构建、测试与诊断工具

至少掌握：编译警告、Debug/Release 区别、CMake target、单元测试和 Sanitizer。

```bash
clang++ -std=c++17 -Wall -Wextra -Wpedantic demo.cpp
clang++ -std=c++17 -fsanitize=address,undefined -g demo.cpp
```

警告不是证明，但高质量项目应解释并尽量消除警告。测试验证已知行为，AddressSanitizer/UndefinedBehaviorSanitizer 用于发现测试执行路径上的内存与 UB 问题；两者不能替代代码推理。
